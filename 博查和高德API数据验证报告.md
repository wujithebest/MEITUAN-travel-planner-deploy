#### 测试 3：精确引号搜索

**查询**：`"申梦上荷" 点评 人均 推荐菜 口味 环境`

返回 5 条结果，全部来自 city8 和工商信息，**零 UGC**。

**结论**：BoCha 对精确短语匹配会大幅缩小结果范围，反而丢失 UGC 内容。

### 1.3 博查总结

| 能力 | 评分 |
|------|------|
| 精确名称→地址信息 | ✅ 可用 |
| 精确名称→UGC 评价 | ❌ 基本无 |
| 加 UGC 关键词后→评价文本 | ⚠️ 有，但来自其他分店/无关餐厅 |
| 区分归属的能力 | ❌ 需要 NLP 后处理 |
| 适合用途 | 宏观主题召回、攻略文本收集 |

### 2.4 高德评分数据结构

高德 `biz_ext` 返回的评分字段详情：

| 字段 | 含义 | 格式 | 取值 |
|------|------|------|------|
| `rating` | 综合评分 | 字符串 | "4.6"（满分 5.0） |
| `cost` | 人均消费 | 字符串（元） | "103.00" |
| `open_time` | 营业时间 | 字符串 | "10:30-14:00" |
| `opentime2` | 营业时间（详细） | 字符串 | "周一至周日 10:30-14:00" |
| `meal_ordering` | 是否支持点餐 | "0"/"1" | "0" |

**注意**：`featured_reviews`（精选评论）和 `featured_reviews_remake`（重构评论）**始终为空数组**。高德公开 API **不返回文本型 UGC 评价**，只返回聚合后的数值评分。

## 三、对比总结

| 维度 | BoCha（通用搜索） | 高德 POI API (extensions=all) |
|------|-------------------|-------------------------------|
| 精确 POI 匹配 | 1/5（其余为无关结果） | 21/21（全部正确分店） |
| 评分数据 | ❌ 无 | ✅ 4.6 / 5.0 |
| 人均消费 | ❌ 无 | ✅ ¥103 |
| 营业时间 | ❌ 无 | ✅ |
| 电话 | ❌ 无 | ✅ 2个号码 |
| 照片 | ❌ 无 | ✅ 3张 |
| 用户标签 | ❌ 无 | ✅ "杭帮菜/本帮菜/停车方便/双人餐" |
| 文本评价 | ⚠️ 能抓到大众点评片段（其它分店） | ❌ API 不返回文本 |
| 推荐菜 | 360地图有（其它分店） | ❌ 不返回 |
| 直接可用性 | 低（需清洗归类） | 高（结构化数据） |
| 延迟 | ~1-3s | ~200ms |
| 成本 | 有限制（1000次/天免费） | 有限制（个人开发者3万次/天） |

## 四、建议

### 餐饮 POI 数据获取策略

1. **评分/人均/照片/电话/营业时间** → **高德 `extensions=all`**  
   直接可用，结构化，无需清洗。

2. **文本型 UGC（评价/推荐菜）** → **BoCha 补充搜索**  
   查询格式：`{完整店名} 点评 口味 推荐菜`  
   注意：**不拆词**，不加引号，加 `点评` 关键词。  
   结果需要 NLP 清洗区分归属（匹配店名/地址过滤）。

3. **当前项目中**：
   - `step2_macro.py` 的 `_search_place` → 高德 text search + extensions=all 已可用
   - `step2_macro.py` 的 `_enrich_places` → Bocha batch search 只拿到 city8 地址信息
   - 建议：对餐饮 POI 额外调用高德 detail API 补充 `biz_ext.rating` 和 `biz_ext.cost`
