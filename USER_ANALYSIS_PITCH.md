# 用户分析：谁需要“言途”？

## 一页版结构

### 1. 核心用户画像

| 用户身份 | 典型状态 | 主要压力 |
|---|---|---|
| 城市年轻用户 | 周末、下班后想临时出门 | 不想做攻略，希望马上有路线 |
| 本地接待者 | 朋友来访，需要安排城市游玩 | 既要有代表性，又不能踩雷 |
| 同行组织者 | 情侣、朋友、同学小团体出行 | 需要平衡不同人的口味和偏好 |
| 主题体验用户 | 想拍照、文艺、夜景、本地生活 | 有感觉，但说不清具体去哪 |

### 2. 共同痛点

用户不是缺少地点信息，而是缺少把信息变成路线的决策编排。

- 信息分散：攻略、地图、点评、社交内容割裂。
- 决策复杂：去哪、怎么走、吃什么、顺不顺路都要自己判断。
- 偏好难融合：个人口味、同行需求、预算、时间常常冲突。
- 结果不可执行：很多推荐停留在文字建议，不能直接走。

### 3. 真实需求

用户需要的不是更多信息，而是针对四类痛点的完整解决方案：

| 对应痛点 | 用户真实需求 |
|---|---|
| 信息分散 | 一句话获得地点、饮食、交通和路线所需的信息，不必反复切换多个应用。 |
| 决策复杂 | 系统替用户完成“去哪、先后顺序、怎么走、何时吃”的路线编排。 |
| 偏好难融合 | 同时理解个人偏好、同行需求、预算和时间，并在冲突时给出可接受的折中方案。 |
| 结果不可执行 | 输出有明确地点、顺序、时段和地图路径的路线，能够直接出发，并支持继续修改。 |

> 核心需求：把“我想去……”直接变成一条符合我、走得通、还能调整的路线。

### 4. 产品回应

```text
自然语言输入
  -> 用户画像注入
  -> UGC 语义增强
  -> 路线决策编排
  -> 可行性校验
  -> 可执行路线结果
```

### 5. PPT 主结论

**言途解决的不是“找地点”问题，而是城市出行中的“决策编排”问题。**

它把用户模糊的自然语言、个人偏好和网络 UGC 信息，转化为一条个性化、可执行、可持续调整的本地生活路线。
