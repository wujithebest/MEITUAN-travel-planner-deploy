# 自然语言到可执行路线规划结果：算法链路说明

## 0. 链路总览

系统将用户的一句自然语言请求，逐步转换为可在地图上渲染、可替换 POI、可继续对话调整的路线结果。

```text
自然语言输入
  -> 前端会话封装
  -> 对话调度
  -> Step1 意图结构化
  -> Step2 宏观路线骨架生成
  -> Step3 微观 POI 与动线规划
  -> PlanReality 现实可行性校验
  -> Step4 结果封装
  -> 前端地图与路线栏渲染
```

## 1. 输入层：自然语言请求

### 1.1 用户输入

输入可以是单轮请求，也可以是基于上一条路线的继续修改。

示例：

```text
帮我推荐一条适合拍照的文艺路线，有咖啡馆和特色小店，节奏轻松一点
```

### 1.2 前端补充上下文

前端会把自然语言与用户画像、出发地、预算、偏好、当前路线状态一起提交给后端。

主要信息包括：

| 类型 | 内容 |
|---|---|
| 用户画像 | 昵称、年龄、预算、饮食偏好、活动偏好 |
| 空间信息 | 当前城市、路线出发点、设备定位或手动地址 |
| 时间信息 | 当前时间、出行日期、上午/下午/晚上等时间约束 |
| 历史上下文 | 上一条路线、已选 POI、替换/删除/追加操作 |
| 原始请求 | 用户自然语言文本 |

## 2. 对话调度层：判断本轮任务类型

后端首先判断本轮请求属于哪类任务。

| 类型 | 说明 |
|---|---|
| new_plan | 生成一条全新路线 |
| refine_current | 基于已有路线继续调整 |
| answer_only | 只回答问题，不生成路线 |
| utility_fast_path | 厕所、附近吃饭等轻量即时需求 |

调度结果决定后续是否完整执行 Step1-Step4，还是走更快的局部处理逻辑。

## 3. Step1：自然语言意图结构化

Step1 的目标是把模糊的自然语言转换成机器可执行的结构化意图。

### 3.1 LLM 语义解析

模型从用户输入中抽取核心约束：

| 字段 | 示例 |
|---|---|
| 城市 | 北京、上海 |
| 天数 | 一日游、两日游、半日游 |
| 固定地点 | 天安门、故宫、北海公园 |
| 活动主题 | 拍照、文艺、亲子、夜景、骑行 |
| 餐饮需求 | 北京菜、清淡素食、火锅、咖啡 |
| 交通约束 | 只坐地铁、骑行、公交、步行距离限制 |
| 时间约束 | 上午、中午、下午、晚上、下班后 |
| 预算约束 | 人均 150、免费路线、高预算 |

### 3.2 规则修正与兜底

LLM 解析后，系统会用规则层修正容易出错的场景。

典型修正包括：

- 固定 POI 别名归一，例如“环球影城”归一为“北京环球度假区”。
- 将“拍照、咖啡馆、特色小店、节奏轻松”识别为组合主题。
- 将“我不吃辣但朋友想吃川菜”识别为冲突饮食偏好。
- 将“只坐地铁”和“步行不超过 5 分钟”识别为交通与步行双重约束。
- 将多轮追问合并到上一条路线上下文中。

### 3.3 Step1 输出

Step1 输出的是 `ParsedIntent`，作为后续规划的统一输入。

核心结果：

```text
结构化意图 =
城市 + 时间 + 预算 + 固定地点 + 主题标签 + 餐饮标签 + 交通约束 + 路线模式
```

## 4. Step2：宏观路线骨架生成

Step2 的目标是先决定“去哪一片区域、哪些主锚点值得去”。

### 4.1 搜索中心确定

系统会优先确定搜索中心：

1. 用户手动设置的路线出发点。
2. 用户明确提到的地点。
3. 当前城市中心或设备定位。
4. 上一轮路线中的上下文位置。

### 4.2 宏观 POI 召回

系统调用地图搜索与内部规则，召回候选地点。

召回来源包括：

- 固定地点搜索。
- 主题关键词搜索。
- 餐饮关键词搜索。
- 周边兴趣点搜索。
- 特殊场景兜底搜索。

例如“文艺 + 拍照 + 咖啡馆 + 特色小店”，会同时召回：

- 文艺街区。
- 咖啡馆。
- 特色小店。
- 拍照点。
- 轻松散步区域。

### 4.3 候选过滤与评分

候选 POI 会经过多层筛选：

| 维度 | 作用 |
|---|---|
| 城市匹配 | 避免北京请求搜到上海 POI |
| 距离约束 | 避免路线突然跳到很远的位置 |
| 预算约束 | 过滤明显不符合预算的地点 |
| 主题匹配 | 保留与用户意图相关的地点 |
| 时间容量 | 避免半日游塞入过多地点 |
| 固定地点保护 | 用户明确点名的地点优先保留 |

### 4.4 Step2 输出

Step2 输出的是宏观路线骨架 `CompletePlan`。

它包含：

- 推荐区域。
- 主锚点 POI。
- 每天或每个时段的活动方向。
- 后续微观规划所需的搜索关键词。

## 5. Step3：微观 POI 与动线规划

Step3 的目标是把宏观骨架细化为可走、可吃、可导航的路线。

### 5.1 锚点拆解

系统会把一个大区域拆解为多个具体停留点。

例如：

```text
外滩区域
  -> 乍浦路桥
  -> 外白渡桥
  -> 外滩源
  -> 外滩观景平台
```

### 5.2 餐饮点插入

如果用户有餐饮需求，系统会在合适的时间段插入餐饮 POI。

处理逻辑：

- 中午优先插入午餐。
- 晚上优先插入晚餐。
- 餐厅位置需要靠近当前路线。
- 冲突口味场景优先寻找折中方案。
- 如果餐饮不是主任务，不允许餐厅挤掉全部景点。

### 5.3 动线排序

系统会根据距离、时段和主题节奏对 POI 排序。

排序目标：

- 少绕路。
- 时间顺序自然。
- 上午、下午、晚上节奏清楚。
- 固定地点不丢失。
- 交通方式与用户要求一致。

### 5.4 路段生成

系统为相邻 POI 生成路线段 `RouteSegment`。

每段包含：

- 起点。
- 终点。
- 距离。
- 耗时。
- 交通方式。
- 地图渲染所需坐标。

### 5.5 备选池生成

系统会同时生成 `candidate_points`，用于前端替换或增加 POI。

备选点需要满足：

- 与当前路线距离不能过远。
- 与用户主题相关。
- 不重复当前已选 POI。
- 可用于替换同类地点。

## 6. PlanReality：现实可行性校验

路线生成后，系统会检查它是否真的满足用户意图。

主要校验项：

| 校验项 | 说明 |
|---|---|
| 主点数量 | 避免只生成 1 个 POI 的路线 |
| 固定地点 | 用户明确要求的 POI 不能丢 |
| 餐饮占比 | 避免餐厅覆盖全部行程 |
| 主题覆盖 | 拍照、文艺、夜景等主题要有证据 |
| 距离合理性 | 避免跨城、跨区乱跳 |
| 时间容量 | 避免半日游或一日游安排过满 |
| 交通约束 | 地铁、步行、骑行等要求需被尊重 |

如果校验失败，系统会尝试补点、降级或给出可理解的失败原因。

## 7. Early Route：提前渲染机制

为了减少用户等待，系统在 Step3 完成后会先返回一版可渲染路线。

此时前端可以先显示：

- 地图线路。
- 右侧 POI 列表。
- 路线基础信息。
- 备选点池。

Step4 的推荐理由与最终文案可以稍后补齐。

这样可以避免用户一直停留在加载状态。

## 8. Step4：最终结果封装

Step4 将路线转换为前端可直接使用的结果。

### 8.1 输出内容

最终结果包括：

| 字段 | 作用 |
|---|---|
| full_plan | 面向用户阅读的完整行程 |
| route_data.points | 右侧路线栏 POI |
| route_data.segments | 地图路线段 |
| route_data.candidate_points | 替换/增加 POI 的备选池 |
| route_recommend_reason | “为什么推荐”文案 |
| waypoint_annotations | POI 标签、时段、命中偏好 |
| planning_state | 路线状态与后续修改上下文 |

### 8.2 推荐理由生成

推荐理由会结合：

- 用户自然语言关键词。
- 用户口味和活动偏好。
- 路线实际 POI。
- 上午/中餐/下午/晚上等结构。

目标不是简单复述地点，而是说明“为什么这条路线适合这个用户”。

## 9. 前端渲染与交互

前端接收 `route_data` 后，将路线拆成两个主要视图。

### 9.1 地图视图

地图负责展示：

- POI 点位。
- 路线连线。
- 交通方式。
- 路段悬停效果。
- 当前高亮路线。

### 9.2 路线栏视图

右侧路线栏负责展示：

- POI 图片、名称、评分、距离。
- 用户偏好命中标签。
- 替换、增加、删除等操作。
- 备选 POI。
- 路线推荐理由。

### 9.3 多轮修改

用户可以继续输入：

```text
把烤鸭换成涮羊肉，人均 150 以内
```

系统会复用上一轮路线状态，重新判断是否局部修改或完整重规划。

## 10. 算法链路的核心优势

### 10.1 从文本到结构

系统不是直接让大模型输出路线，而是先把自然语言解析为结构化意图，降低不可控性。

### 10.2 从召回到校验

路线不是一次生成完成，而是经过 POI 召回、筛选、排序、插餐、动线、现实校验等多个步骤。

### 10.3 从静态路线到可交互路线

最终结果不仅是一段文本，而是可在地图上展示、可替换、可删除、可继续对话调整的路线数据。

### 10.4 从完整等待到提前渲染

系统支持 Step3 后提前返回可视化路线，Step4 再补充最终说明，提高前端响应速度。

## 11. 一句话总结

本系统将用户自然语言请求“编译”为结构化意图，再通过地点召回、路线组合、现实校验和前端渲染，生成一条可执行、可解释、可交互调整的本地生活路线。
